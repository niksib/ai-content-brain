# Epic: Agents memory update (memory-map + on-demand read)

## Концепт

После завершения эпика онбординга у юзера есть `MemoryBlock` таблица с каноническими ключами. Агенты сейчас работают по-старому: `loadContext()` грузит весь профиль в системный промпт одним объектом. Задача — перевести агентов на **memory-map + read tool** архитектуру.

## Цель

1. **Дешевле context** — не тащим весь профиль в каждый LLM-вызов, только то что нужно под задачу
2. **Масштабируется** — по мере роста memory (через epic memory-write) контекст не раздувается
3. **Прозрачнее для агента** — он сам решает что прочитать под запрос юзера

## Как работает

### First session init
При старте чат-сессии контекст-билдер агента собирает:
- **Memory map** — список всех блоков юзера в формате `{ key, title, description }` (БЕЗ content)
- Системный промпт + инструкция: "У тебя есть инструмент `read_memory(keys[])`. Используй его чтобы подгрузить нужные блоки под задачу юзера"

### Read tool
Агент вызывает `read_memory(keys: string[])` → возвращаются полные `content` запрошенных блоков. Результат попадает в tool result, агент продолжает с этим контекстом.

### Пример
Юзер: "Сделай пост про мой новый проект"
→ Агент читает memory map, видит блок `current_projects`
→ Вызывает `read_memory(["current_projects", "voice_tone", "content_pillars"])`
→ Получает полные содержимые
→ Генерит пост

## Scope

- `StrategistAgent` уже переведён в эпике онбординга (если выбран вариант c из обсуждения)
- Переводим: `ThreadsAgent`, `InstagramAgent`, `LinkedInAgent`, `VideoAgent`
- Удаляем адаптер "собрать всё в один объект" из platform agents
- Обновляем системные промпты — инструкции по работе с memory-map и read tool

## Зависимости

- **Требует**: завершённый epic онбординга с `MemoryBlock` схемой
- **Блокирует**: epic memory-write (ему нужен read tool для конфликт-handling)

## Открытые вопросы

- Кэшировать ли результат `read_memory` внутри одной сессии (чтобы не читать тот же блок дважды)?
- Логировать вызовы read tool для аналитики — какие блоки используются реально, какие мёртвые?
- Стоит ли давать агенту pre-fetch самые частые блоки (`niche`, `voice_tone`) без tool-вызова, а остальные через tool?
